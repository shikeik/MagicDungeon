## Gemini
PostgreSQL（Supabase 的底层数据库）本身不像 Redis 那样有原生的“TTL（过期时间）”功能来让数据自动消失。

要实现“超时30秒自动删除”，通常有两种做法结合使用：**物理删除**（后台清理）和 **逻辑隐藏**（前端过滤）。

以下是具体方案：

### 方案一：使用 `pg_cron` 定时任务（推荐，自动后台清理）

Supabase 自带一个强大的扩展叫 `pg_cron`，可以设置定时任务。虽然 Cron 最小单位是**每分钟**执行一次，但这通常用于“垃圾回收”已经足够了。

**步骤：**

1.  打开 Supabase 左侧菜单的 **SQL Editor**。
2.  运行以下 SQL 代码来启用扩展并创建任务：

```sql
-- 1. 启用 pg_cron 扩展（如果尚未启用）
create extension if not exists pg_cron;

-- 2. 创建一个定时任务：每分钟执行一次清理命令
-- 注意：Cron 最快只能每分钟跑一次，所以数据可能会在数据库里多留几十秒，但这不影响业务逻辑（见方案二）
select cron.schedule(
  'delete-old-rooms',             -- 任务名称
  '* * * * *',                    -- Cron 表达式 (每分钟)
  $$ DELETE FROM rooms WHERE last_ping < now() - INTERVAL '30 seconds' $$ -- 执行的 SQL
);
```

**如何查看或停止任务？**

*   查看任务：`select * from cron.job;`
*   删除任务：`select cron.unschedule('delete-old-rooms');`

---

### 方案二：视图/前端过滤（关键，实现“立即”过期的效果）

因为 `pg_cron` 最快也是每分钟删一次，所以在第31秒到第59秒之间，数据可能还在库里。为了让用户感觉是“严格30秒超时”，你必须在**读取数据**时就把超时的数据过滤掉。

**做法 A：在前端查询时过滤**
不要直接 `select * from rooms`，而是带上时间条件：

```javascript
// 前端代码示例 (Supabase JS SDK)
const { data, error } = await supabase
  .from('rooms')
  .select('*')
  .gt('last_ping', new Date(Date.now() - 30 * 1000).toISOString()) // 只查最近30秒心跳的
```

**做法 B：创建一个数据库 View (视图)**
创建一个“只显示活跃房间”的视图，前端只查这个视图：

```sql
create view active_rooms as
select *
from rooms
where last_ping > now() - interval '30 seconds';
```
以后前端直接查 `active_rooms`，过期的房间绝对不会出现，而后台的 `pg_cron` 会负责慢慢清理物理数据释放空间。

### 方案三：针对游戏开发的建议 (Realtime Presence)

看你的表结构 (`host_port`, `player_count`, `rooms`)，你似乎是在做一个**多人游戏的联机大厅 (Lobby)**。

对于这种场景，写数据库其实不是最高效的（因为需要不断 update `last_ping`，数据库压力大）。Supabase 有一个专门针对这种“在线状态”的功能叫 **Supabase Realtime Presence**。

**它的原理是：**
1.  客户端连接 WebSocket。
2.  告诉服务器“我在线”。
3.  如果客户端断网、崩溃或关闭页面，Supabase 会自动检测到并将其从列表移除（几乎是实时的）。
4.  不需要存数据库，不需要手动删行。

如果你的业务允许不存入硬盘，只在内存中维护在线列表，强烈建议研究一下 **Supabase Presence** 文档。

### 什么是 Presence？

想象一下，你现在的做法（SQL 方案）是：
> 每个人每隔几秒钟就在黑板（数据库）上写字：“我还活着”，然后你雇了一个清洁工（pg_cron）每分钟去擦掉那些很久没更新的名字。

而 **Presence** 的做法是：
> 大家走进一个房间（WebSocket 通道），互相都能看到对方。如果一个人走出了房间（断网或关闭网页），他就自动从大家的视野里消失了。

**Presence 是纯内存的、实时的状态同步机制。** 它不写硬盘数据库，专门用来解决“谁在线”、“谁在哪个房间”这个问题。

### 为什么它适合你的游戏大厅？

1.  **自动处理“非正常断线”：**
	*   **SQL 方案：** 如果玩家突然拔网线，数据库里他的 `last_ping` 还在，必须等 30 秒超时脚本跑完，别人才会觉得他下线了。
	*   **Presence 方案：** 玩家断开 WebSocket 连接（拔网线或关进程），Supabase 服务器会立即检测到，并马上通知其他所有客户端“玩家 X 已离开”。**几乎是毫秒级反应，不需要等待 30 秒。**

2.  **极低延迟：**
	*   新的玩家加入房间，其他人的屏幕上会瞬间刷出来，不用轮询数据库。

3.  **节省数据库资源：**
	*   你不需要每隔几秒钟就发起一个 `UPDATE rooms SET last_ping = now()...` 请求。你的数据库 CPU 和磁盘 IO 会非常感谢你。

### 代码怎么写？（JavaScript 示例）

假设你要做一个游戏大厅 `lobby`：

```javascript
// 1. 创建一个频道
const channel = supabase.channel('game_lobby')

channel
  .on('presence', { event: 'sync' }, () => {
    // 每当有人加入或离开，这里都会触发
    // 获取当前所有在线玩家列表
    const newState = channel.presenceState()
    console.log('当前在线玩家:', newState)
    // 这里更新你的 UI 列表
  })
  .on('presence', { event: 'join' }, ({ key, newPresences }) => {
    console.log('有人进来了:', newPresences)
  })
  .on('presence', { event: 'leave' }, ({ key, leftPresences }) => {
    console.log('有人走了:', leftPresences)
  })
  .subscribe(async (status) => {
    if (status === 'SUBSCRIBED') {
      // 2. 连接成功后，上传我的信息（比如房间号、玩家名）
      await channel.track({
        user_id: 'player_123',
        room_name: '新手村',
        level: 5
      })
    }
  })
```

就是这么简单。你不需要写任何后端代码，也不需要配置 Cron 任务。

### 关于免费额度（Free Tier）

虽然免费，但有配额限制（防止被滥用）。根据 Supabase 官网目前的定价：

| 指标 | 免费版 (Free) | 专业版 (Pro) | 说明 |
| :--- | :--- | :--- | :--- |
| **并发连接数** | **200 个** | 500 个 (可付费增加) | 同时在线的玩家数量。如果你的游戏同时有超过 200 人在线，说明你该赚钱升级了。 |
| **消息数量** | **200 万条/月** | 500 万条/月 | Presence 的状态同步也是消息。 |
| **Presence 限制** | 包含在内 | 包含在内 | 主要是并发数的限制最关键。 |

**总结：**
如果你的游戏还在开发中，或者同时在线人数不超过 200 人，**Presence 是完全免费且性能最好的选择**。

如果你的游戏以后火了，几千人在线，那时候升级到 Pro 套餐（$25/月）也是非常划算的，因为它帮你省下了大量的数据库维护成本。
