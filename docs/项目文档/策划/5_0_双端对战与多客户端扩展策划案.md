### 1.2 已验证能力

| 能力 | 状态 | 说明 |
|------|------|------|
| 状态同步 (NetworkVariable) | ✅ | Server tick → 自动广播脏变量 |
| Spawn 自动派生 | ✅ | spawnWithPrefab → Client 工厂创建副本 |
| Despawn 网络销毁 | ✅ | 0x12 封包双端清理 |
| ClientRpc (S→C) | ✅ | 反射分发，子弹生成验证通过 |
| ServerRpc (C→S) | ✅ | 框架已实现，沙盒尚未使用 |
| UDP 跨进程 | ✅ | 握手 + 长度前缀 + 守护线程 |
| 多客户端网络层 | ✅ | UdpSocketTransport 已支持多 Client 地址列表 |

### 1.3 瓶颈与不足

| # | 问题 | 影响 | 严重度 |
|---|------|------|--------|
| 1 | **Client 无法操作** | UDP 客户端是纯只读观察模式，不能发送输入 | 🔴 |
| 2 | **IP/Port 硬编码** | `127.0.0.1:19100` 写死，无法局域网/广域网测试 | 🔴 |
| 3 | **固定双坦克** | Server 写死 Spawn 2 辆坦克，不能动态加入 | 🟡 |
| 4 | **TransportReceiveCallback 无 clientId** | Server 收到 RPC 包不知道来自哪个 Client | 🔴 |
| 5 | **无连接事件回调** | Server 不知道新 Client 何时连入 / 断线 | 🟡 |
| 6 | **碰撞检测只判 ownerId=1/2** | 写死两个坦克的碰撞逻辑，不支持 N 个坦克 | 🟡 |

### 2.1 用户可感知目标

| # | 目标 | 玩家感知 |

|---|------|------|
| A | **TransportReceiveCallback 携带 clientId** | Server 收包时能区分来源 |

## 三、框架通用性评估（回答问题 4）

> 用户问："现在框架扩展性怎么样，不会只能用于坦克吧？"

### 3.1 结论：**框架本身是通用的，瓶颈在沙盒层硬编码**

| 层 | 通用性 | 说明 |
|----|--------|------|
| NetworkManager | ✅ 通用 | 不含任何游戏逻辑，纯粹管理 Spawn/Despawn/Tick/RPC 分发 |
| NetworkVariable\<T\> | ✅ 通用 | 支持 int/float/boolean/String/Color，可扩展 |
| NetworkBehaviour | ✅ 通用 | 业务继承它即可，反射自动发现变量和 RPC |
| Transport 接口 | ✅ 通用 | LocalMemory / UDP 可插拔，未来可接 TCP / WebSocket |
| NetBuffer | ✅ 通用 | 通用序列化工具 |

**瓶颈全在沙盒层**：

| 硬编码 | 位置 | 修复方式 |
|--------|------|----------|
| `ownerId=1/2` 碰撞判断 | UdpServerScreen / TankSandboxScreen | 改为遍历所有坦克列表 |
| 固定 Spawn 2 辆 | onClientConnected() | 改为每 Client 连入时动态 Spawn |
| 客户端不能操作 | 无 ServerRpc 输入通道 | TankBehaviour 增加 rpcInput |
| IP 硬编码 | 常量 SERVER_IP / PORT | 改为 UI 输入 |

### 3.2 将来应用于魔法地牢时需要做的

框架本身不需要改。业务层只需要：
1. 写一个 `PlayerBehaviour extends NetworkBehaviour`（带 position、hp、action 等 NetworkVariable）
2. 写一个 `EnemyBehaviour extends NetworkBehaviour`（敌人由 Server 控制 AI）
3. Server 端 Spawn 地图种子实体 + Spawn 敌人 + 每个 Client 连入时 Spawn 一个玩家
4. 客户端通过 `@ServerRpc` 上传移动/攻击输入

## 四、详细改动方案

### Phase 1：引擎层 —— TransportReceiveCallback 携带 clientId

**目标**：Server 收到 Client 的数据时，知道是哪个 Client 发来的。

**改动清单**：

#### 4.1.1 TransportReceiveCallback 接口

```java
// 改前
@FunctionalInterface
public interface TransportReceiveCallback {
    void onReceiveData(byte[] payload);
}

// 改后
@FunctionalInterface
public interface TransportReceiveCallback {
    /**
     * @param payload  原始字节数据
     * @param clientId 发送方的 clientId（Server 端接收时有意义，Client 端固定为 -1）
     */
    void onReceiveData(byte[] payload, int clientId);
}
```

#### 4.1.2 UdpSocketTransport

- `processReceivedData()` 中根据 `sender` 地址查 `clientAddresses` 列表的 index 得到 clientId
- 回调改为 `receiveCallback.onReceiveData(payload, clientId)`

#### 4.1.3 LocalMemoryTransport

- Server 端收到的 clientId 固定为 0（只有一个模拟 Client）
- Client 端收到的 clientId 固定为 -1（来自 Server）

#### 4.1.4 NetworkManager.onReceiveData

- 方法签名改为 `onReceiveData(byte[] payload, int clientId)`
- setTransport 注册回调适配新签名
- ServerRpc 处理时可用 clientId 做权限校验

---

### Phase 2：引擎层 —— 连接事件回调

**目标**：Server 知道 Client 何时连入 / 断开。

**改动方式**：在 NetworkManager 增加连接监听器接口。

```java
public interface NetworkConnectionListener {
    void onClientConnected(int clientId);
    void onClientDisconnected(int clientId);    // 暂留，后续心跳时实现
}
```

- UdpSocketTransport 在 `processReceivedData()` 发现新客户端时，除了加入列表，还回调 `onClientConnected`
- NetworkManager 持有 listener 引用，收到回调后由业务层处理（如 Spawn 坦克）

---

### Phase 3：引擎层 —— NetworkObject 增加 ownerClientId

```java
public class NetworkObject {
    private int ownerClientId = -1;  // -1 表示 Server 拥有
    // getter / setter
}
```

- `spawnWithPrefab()` 新增重载：`spawnWithPrefab(int prefabId, int ownerClientId)`
- SpawnPacket 格式扩展：`[0x11][netId][prefabId][ownerClientId]`
- Client 收到 SpawnPacket 后，如果 `ownerClientId == 自己的 clientId`，设 `isLocalPlayer = true`

> **注意**：Client 需要知道自己的 clientId。方案：Server 在握手回复中告知 Client 分配的 clientId。

---

### Phase 4：沙盒层 —— TankBehaviour 增加 ServerRpc 输入上行

```java
public class TankBehaviour extends NetworkBehaviour {
    // ... 现有 NetworkVariable ...

    /** Client → Server: 上报输入 */
    @ServerRpc
    public void rpcInput(float dx, float dy, boolean fire) {
        // Server 端执行：此方法体在 Server 端运行
        // dx/dy 归一化方向，fire 是否开火
    }
}
```

- Server 收到 rpcInput 后，根据 ownerClientId 匹配坦克，执行移动/开火逻辑
- Client 每帧检测键盘/手柄输入，打包为 `sendServerRpc("rpcInput", dx, dy, fire)`

### Phase 5：沙盒层 —— 统一坦克联机屏幕 + 配置 UI

**合并 NetcodeUdpServerScreen + NetcodeUdpClientScreen → NetcodeTankOnlineScreen**

#### 5.1 配置 UI 面板

进入屏幕时显示选择面板（用 VisUI 或 NeonBatch 简易绘制）：

```
╔══════════════════════════════════╗
║      Netcode 坦克联机对战         ║
║                                  ║
║  [创建房间 (Server)]              ║
║    监听端口: [19100]              ║
║                                  ║
║  [加入房间 (Client)]              ║
║    服务器IP: [192.168.1.100]      ║
║    端口:     [19100]              ║
║                                  ║
║  [开始]                           ║
╚══════════════════════════════════╝
```

- 选「创建房间」→ 输入端口 → 启动 Server，等待客户端
- 选「加入房间」→ 输入 IP + 端口 → 连接 Server
- 局域网测试：IP 填同网段地址
- 广域网测试：IP 填公网 IP（需手动端口转发）

#### 5.2 状态机

```
CONFIG → (用户点击开始) → WAITING → (条件满足) → PLAYING → (断线) → DISCONNECTED
```

| 状态 | Server 行为 | Client 行为 |
|------|-------------|-------------|
| CONFIG | 显示 UI 面板 | 显示 UI 面板 |
| WAITING | 监听端口，每有 Client 连入就 Spawn 一辆坦克 | 连接 Server，轮询 SpawnPacket |
| PLAYING | 处理输入RPC + 物理 + tick | 读取输入 → sendServerRpc + 渲染同步状态 |
| DISCONNECTED | 提示 Client 断开 | 提示连接断开 |

### Phase 6：沙盒层 —— 多客户端动态 Spawn

#### 6.1 Server 端

```java
// 连接事件：每个 Client 连入时
void onClientConnected(int clientId) {
    NetworkObject obj = manager.spawnWithPrefab(TANK_PREFAB_ID, clientId);
    TankBehaviour tank = (TankBehaviour) obj.getBehaviours().get(0);
    // 随机颜色 + 随机出生点
    tank.x.setValue(randomSpawnX());
    tank.y.setValue(randomSpawnY());
    tank.color.setValue(randomColor(clientId));
    // 记录映射
    clientTanks.put(clientId, tank);
}
```

- Server 自身也可以产一辆坦克（Host 模式，clientId = -1 代表本机）
- 碰撞检测改为遍历 `clientTanks.values()` 所有坦克

#### 6.2 Client 端

- 无需知道具体坦克数量，所有实体由 SpawnPacket 自动派生
- 遍历 `manager.getAllNetworkObjects()` 提取 TankBehaviour 渲染
- 自己的坦克通过 `isLocalPlayer == true` 识别并发送输入

## 五、碰撞检测泛化

当前碰撞逻辑硬编码了 `ownerId != 1` / `ownerId != 2`，改为泛化：

```java
// 改前：
if (b.ownerId != 1 && !p1.isDead.getValue() && distance(b, p1) < 20) { hitTank(p1); }
if (b.ownerId != 2 && !p2.isDead.getValue() && distance(b, p2) < 20) { hitTank(p2); }

// 改后：
for (Map.Entry<Integer, TankBehaviour> entry : clientTanks.entrySet()) {
    int tankOwnerId = entry.getKey();
    TankBehaviour tank = entry.getValue();
    if (b.ownerId != tankOwnerId && !tank.isDead.getValue() 
        && distance(b, tank) < 20) {
        TankSandboxUtils.hitTank(tank);
        iter.remove();
        break;
    }
}
```

## 六、握手协议升级：分配 clientId

当前握手：Client 发 `[0xFF×4]` → Server 记录地址。  
升级：Server 回复 `[0xFE×4][4字节 clientId]`，Client 收到后缓存自己的 clientId。

```
Client                          Server
  │─── [0xFF,0xFF,0xFF,0xFF] ──→│  握手
  │                              │  记录 Client 地址，分配 clientId=N
  │←── [0xFE,0xFE,0xFE,0xFE][N]│  握手回复
  │                              │
  │←── SpawnPacket(坦克)         │  为该 Client Spawn 坦克 (ownerClientId=N)
```

- UdpSocketTransport 新增 `HANDSHAKE_REPLY_MAGIC = [0xFE×4]`
- Client 收到握手回复后存储 `myClientId`
- NetworkManager 新增 `setLocalClientId(int id)`，Client 设置后用于判断 `isLocalPlayer`

## 九、风险与注意事项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| UDP 丢包导致 SpawnPacket 丢失 | Client 少一辆坦克 | 后续增加可靠传输（序号+重传），当前先用回环测试 |
| 广域网延迟导致操作迟钝 | 手感差 | 后续可加客户端预测（Client Prediction），当前先验证基础链路 |
| RPC 参数序列化不支持更多类型 | 复杂游戏需要 Vector2/自定义类型 | 后续扩展 NetBuffer |
| 多线程竞争（UDP 收包线程 vs 主线程） | 偶现 ConcurrentModification | 收包缓存到队列，主线程 tick 时消费 |

## 十、验收标准

1. **✅ 双端对战**：两个进程各操控一辆坦克互相射击，命中扣血，死亡复活
2. **✅ IP 可配**：UI 面板输入局域网 IP 后能跨机器联机
3. **✅ 多客户端**：3 个以上客户端同时连入，各自操控独立坦克
4. **✅ 向后兼容**：纯内存沙盒仍能正常运行，32 个测试用例全绿
5. **✅ 零硬编码**：碰撞检测、坦克数量均由运行时数据驱动
