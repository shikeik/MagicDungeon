# 9_0 坦克联机屏重构解耦策划案

## 1. 问题概述

当前 `NetcodeTankOnlineScreen.java` 共 **1116 行**，是典型的 **"上帝类"(God Class)**，混合了以下 7+ 种职责：

| # | 职责 | 行数范围(约) | 问题 |
|---|------|-------------|------|
| 1 | 预配置 + 配置面板输入 | 40-100, 265-300 | 静态预配置 + UI 逻辑混杂 |
| 2 | 网络会话管理 | 300-360 | startNetwork/shutdownNetwork 与游戏状态耦合 |
| 3 | Server 游戏逻辑 | 400-490 | 坦克输入、移动、死亡复活、tick 同步 |
| 4 | Client 游戏逻辑 | 545-630 | 客户端预测、RPC 上报、子弹模拟 |
| 5 | 子弹物理+碰撞 | 490-540, 600-630 | Server/Client 子弹分散在不同方法 |
| 6 | 地图墙体+碰撞 | 1020-1116 | 墙体生成、边界clamp、推出墙体 |
| 7 | 渲染(配置/等待/游戏/HUD/抽屉/断线) | 715-990 | 6种渲染模式全在一个类 |
| 8 | 连接事件+坦克Spawn | 640-690 | 连接回调与游戏对象管理耦合 |
| 9 | 相机跟随 | 695-710 | 零散内嵌 |

### 核心痛点

1. **无法单元测试**：逻辑与 libGDX 渲染/输入深度耦合，纯逻辑无法脱离图形上下文测试
2. **修一处坏三处**：Server/Client 逻辑交叉引用，碰撞/子弹/坦克状态互相嵌套
3. **可读性极差**：1100+ 行需要在脑中同时维护 7 种状态上下文
4. **扩展困难**：新增帧体/武器类型需要修改多处分散代码

---

## 2. 重构目标

- **职责单一**：每个类只负责一个明确的职责领域
- **可测试性**：核心逻辑类不依赖 libGDX 图形/输入 API，可在 headless 环境下单元测试
- **低耦合**：模块间通过接口/回调通信，替换一个模块不影响其他
- **Screen 变协调器**：`NetcodeTankOnlineScreen` 精简为 ~200 行的组装调度器

---

## 3. 目标架构

```
NetcodeTankOnlineScreen (协调器, ~200行)
 │
 ├── NetworkSession          // 网络会话管理（连接/断开/心跳/配置）
 │
 ├── TankGameMap              // 地图数据与碰撞逻辑（墙体生成、边界、AABB）
 │
 ├── BulletSystem             // 子弹管理（Server权威物理 + Client模拟）
 │
 ├── TankSpawnSystem          // 坦克生成/复活/管理（出生点、颜色分配）
 │
 ├── ServerGameLogic          // Server 端帧逻辑（输入处理、tick驱动）
 │
 ├── ClientGameLogic          // Client 端帧逻辑（预测、RPC上报）
 │
 └── UI 渲染模块
      ├── ConfigPanelRenderer  // 配置界面渲染+输入
      ├── WaitingRenderer      // 等待界面渲染
      ├── GameRenderer         // 游戏世界渲染（坦克+子弹+地图）
      ├── HudRenderer          // HUD + 抽屉面板
      └── DisconnectOverlay    // 断线覆盖层
```

## 4. 各模块详细设计

### 4.1 TankGameMap — 地图数据与碰撞

**文件**: `TankGameMap.java`
**职责**: 墙体生成、边界约束、AABB碰撞检测
**可测试性**: ✅ 纯数据+数学，零图形依赖

```java
public class TankGameMap {
    private final float mapWidth, mapHeight;
    private final List<float[]> walls = new ArrayList<>();

    // 根据种子生成墙体
    public void generateWalls(long seed);

    // 将坐标clamp到边界内
    public Vector2 clampToBoundary(float x, float y, float halfSize);

    // 推出墙体碰撞
    public Vector2 pushOutOfWalls(float x, float y, float halfSize);

    // 子弹是否命中墙体
    public boolean bulletHitsWall(float bx, float by, float bulletSize);

    // 子弹是否越界
    public boolean isOutOfBounds(float x, float y);

    // Getter
    public float getMapWidth();
    public float getMapHeight();
    public List<float[]> getWalls();
}
```

**单元测试要点**:
- 同种子生成一致墙体
- 边界clamp正确
- 墙体推出方向正确
- 子弹墙体碰撞检测

### 4.2 BulletSystem — 子弹管理

**文件**: `BulletSystem.java`
**职责**: Server端子弹物理+坦克碰撞, Client端子弹模拟
**可测试性**: ✅ 依赖 TankGameMap 接口，不依赖图形

```java
public class BulletSystem {
    private final List<Bullet> bullets = new ArrayList<>();
    private int nextBulletId = 1;

    // Server端: 生成子弹
    public Bullet spawnBullet(TankBehaviour tank, int ownerId);

    // Server端: 更新所有子弹物理 + 碰撞检测
    // 返回: 命中事件列表 [(bulletId, hitTankOwnerId)]
    public List<BulletHitEvent> updateServerBullets(float delta, TankGameMap map,
                                                     Map<Integer, TankBehaviour> tanks);

    // Client端: 更新本地子弹模拟（从NetworkObject收集）
    public void updateClientBullets(float delta, TankGameMap map,
                                     Iterable<NetworkObject> networkObjects);

    // Getter
    public List<Bullet> getBullets();
    public int getNextBulletId();
    public void clear();

    /** 子弹命中事件 */
    public static class BulletHitEvent {
        public final int bulletId;
        public final int hitTankOwnerId;
        public final int shooterOwnerId;
    }
}
```

**单元测试要点**:
- 子弹生成位置/速度计算
- 子弹移动物理
- 墙体碰撞消除
- 越界消除
- 坦克碰撞检测（不打自己、跳过死亡坦克）

### 4.3 TankSpawnSystem — 坦克生成管理

**文件**: `TankSpawnSystem.java`
**职责**: 坦克生成/复活/出生点分配/颜色分配
**可测试性**: ✅ 纯逻辑

```java
public class TankSpawnSystem {
    private static final Color[] SPAWN_COLORS = { ... };
    private final Map<Integer, TankBehaviour> tanks = new HashMap<>();

    // 为指定owner Spawn坦克（需要 NetworkManager 生成 NetworkObject）

    // 复活坦克
    public void respawnTank(int ownerClientId);

    // 移除坦克
    public void removeTank(int ownerClientId);

    // 计算出生点

    // 计算出生颜色
    public Color getSpawnColor(int index);

    // Getter
    public Map<Integer, TankBehaviour> getTanks();
    public TankBehaviour getTank(int ownerClientId);
    public int getTankCount();
    public void clear();
```

**单元测试要点**:
- 出生点坐标计算（按索引排列）
- 颜色循环分配
- 复活重置属性
- 添加/移除坦克映射

### 4.4 NetworkSession — 网络会话管理

**文件**: `NetworkSession.java`
**职责**: 网络连接生命周期管理（启动/关闭/心跳/断线检测）
**可测试性**: ⚠️ 部分可测（Mock Transport）

```java
public class NetworkSession {
    public interface ConnectionCallback {
        void onClientConnected(int clientId);
        void onClientDisconnected(int clientId);
    }

    private NetworkManager manager;
    private ReliableUdpTransport transport;
    private UdpSocketTransport rawTransport;
    private boolean isServer;
    private boolean connectionLost;
    private float connectionLostTimer;

    // 启动（Server/Client）
    public void start(boolean isServer, String ip, int port,
                      ConnectionCallback callback) throws Exception;

    // 关闭
    public void shutdown();

    // 每帧调用: 心跳检测+可靠层tick
    public void tick(float delta);

    // 状态查询
    public boolean isConnectionLost();
    public float getConnectionLostTimer();
    public long getPingMs();
    public int getActiveClientCount();
    public NetworkManager getManager();
    public ReliableUdpTransport getTransport();
}
```

### 4.5 ServerGameLogic / ClientGameLogic — 端侧帧逻辑

**文件**: `ServerGameLogic.java`, `ClientGameLogic.java`
**职责**: 分离 Server/Client 每帧的更新逻辑

```java
// ServerGameLogic.java
public class ServerGameLogic {
    /** 每帧更新（Server权威逻辑） */
    public void update(float delta, TankSpawnSystem tanks,
                       BulletSystem bullets, TankGameMap map,
                       NetworkSession session);

    /** 处理Host本地键盘输入 */
    public void processHostInput(TankBehaviour hostTank, float delta);
}

// ClientGameLogic.java
public class ClientGameLogic {
    /** 每帧更新（Client预测+上报） */
    public void update(float delta, BulletSystem bullets,
                       TankGameMap map, NetworkSession session);
}
```

**可测试性**: ⚠️ 输入部分依赖 Gdx.input（可通过接口抽象或 headless mock）

### 4.6 UI 渲染模块

将渲染拆为 5 个独立 Renderer，每个只负责一个画面状态：

| 类名 | 职责 | 依赖数据 |
|------|------|----------|
| `ConfigPanelRenderer` | 配置界面渲染+输入处理 | 配置状态 |
| `WaitingRenderer` | 等待界面渲染 | 连接状态 |
| `GameWorldRenderer` | 游戏世界(坦克+子弹+地图) | 坦克列表、子弹列表、地图 |
| `HudRenderer` | HUD信息+抽屉面板 | 连接信息、坦克列表 |
| `DisconnectOverlayRenderer` | 断线覆盖层 | 断线状态 |

渲染模块由于依赖 NeonBatch/BitmapFont 等图形API，**主要通过集成测试/手动验证**，但通过将数据与渲染分离，渲染代码极度简化，出错概率大幅降低。

### 关键设计决策

1. **`createBullet()` vs `spawnBullet()`**: `createBullet()` 纯数据创建（用于测试），`spawnBullet()` 带 RPC 广播（用于生产）
2. **RPC 空值守护**: 所有 `sendClientRpc/sendServerRpc` 调用前加 `if (getManager() != null)` 判断，确保脱离网络上下文也能运行
3. **TankSpawnSystem 静态工具模式**: 出生/复活逻辑均为 static 方法，坦克映射仍由 Screen 持有
4. **Renderer 持有画笔**: `TankGameRenderer` 内部创建并管理 NeonBatch + BitmapFont，Screen 不再持有图形资源
5. **`Collections.unmodifiableList`**: 客户端子弹列表返回不可变视图，防止外部误修改
