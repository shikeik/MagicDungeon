## 一、问题描述

**现象**: 通过大厅房间列表加入的客户端能正确渲染地图障碍，而通过"直连"按钮加入的客户端看到的墙体位置与服务端完全不同。

- 客户端渲染的是"假墙"——视觉上有墙但不阻挡移动
- 服务端的"真墙"客户端看不见，但逻辑上确实阻挡坦克
- 服务端权威逻辑正确（碰撞检测在 Server 端执行），仅客户端渲染不一致

## 二、根因分析

### 地图种子的三条加入路径

地图障碍由 `TankGameMap.generateWalls(seed)` 根据随机种子生成。双端必须使用相同种子才能得到一致的墙体布局。

修复前，种子由双端各自猜测：

```
Server 端 (startNetwork):
    long mapSeed = roomName != null ? roomName.hashCode() : System.currentTimeMillis();

Client 端 (startNetwork):
    long mapSeed = roomName != null ? roomName.hashCode() : 42;
```

三条路径的种子匹配情况：

| 加入方式 | Host roomName | Client roomName | 种子一致？ |
|----------|--------------|-----------------|-----------|
| 房间列表 | `"xxx 的房间"` | `"xxx 的房间"` (从列表获取) | ✅ 碰巧一致 |
| 大厅直连 | `"xxx 的房间"` | `"直连房间"` (硬编码) | ❌ 不一致 |
| CONFIG 手动输入 | `null` → `currentTimeMillis()` | `null` → `42` | ❌ 不一致 |

**根因**: 地图种子不是由 Server 权威下发的，双端各自猜测种子，只有房间列表路径碰巧匹配。

## 三、修复方案

### 设计原则

> **Server 权威下发，Client 不猜测。**

将地图种子的分发从"双端各自计算"改为 Server 通过 ClientRpc 广播，客户端收到后再生成地图。

### 数据流

```
Server                                    Client
  │                                         │
  │  startNetwork()                         │  startNetwork()
  │  gameMap.generateWalls(seed)            │  (不生成地图，等待种子)
  │                                         │
  │  onNewClientConnected(clientId)         │
  │  ├─ sendExistingSpawns                  │
  │  ├─ sendFullState                       │
  │  ├─ spawnTankForOwner(clientId)         │
  │  ├─ sendFullState (含新坦克)             │
  │  └─ hostTank.sendClientRpc              │
  │       ("rpcSyncMapSeed", seed) ─────────┤  rpcSyncMapSeed(seed)
  │                                         │  ├─ receivedMapSeed = seed
  │                                         │  └─ mapSeedReceived = true
  │                                         │
  │                                         │  checkMapSeedSync() (每帧)
  │                                         │  ├─ 检测 mapSeedReceived
  │                                         │  ├─ gameMap.generateWalls(seed)
  │                                         │  └─ mapSeedReceived = false
  │                                         │
  │          双端墙体布局一致 ✓               │
```

### 代码变更

#### 1. TankBehaviour.java (+17 行)

新增 `@ClientRpc rpcSyncMapSeed(int seed)`:

```java
// IO 线程安全字段
public transient volatile int receivedMapSeed = 0;
public transient volatile boolean mapSeedReceived = false;

@ClientRpc
public void rpcSyncMapSeed(int seed) {
    receivedMapSeed = seed;
    mapSeedReceived = true;
}
```

- 使用 `volatile` 保证 IO 线程（RPC 接收）→ 主线程（渲染帧）的可见性
- 采用 flag + value 的双字段模式，与 `pendingSpawns`/`pendingDestroys` 保持一致的线程安全风格

#### 2. NetcodeTankOnlineScreen.java (+29 / -6 行)

**Server 端** — 种子类型从 `long` 改为 `int`（兼容 RPC int 序列化）:

```java
// 修复前
long mapSeed = roomName != null ? roomName.hashCode() : System.currentTimeMillis();

// 修复后
int mapSeed = roomName != null ? roomName.hashCode() : (int) System.currentTimeMillis();
```

**Server 端** — 新客户端连入时广播种子:

```java
// onNewClientConnected 末尾新增
hostTank.sendClientRpc("rpcSyncMapSeed", (int) gameMap.getMapSeed());
```

**Client 端** — 删除自行猜测种子的代码:

```java
// 修复前
long mapSeed = roomName != null ? roomName.hashCode() : 42;
gameMap.generateWalls(mapSeed);

// 修复后
// (完全删除，等 Server 下发)
```

**Client 端** — 新增 `checkMapSeedSync()` 每帧检查:

```java
private void checkMapSeedSync() {
    for (NetworkObject obj : manager.getAllNetworkObjects()) {
        for (NetworkBehaviour b : obj.getBehaviours()) {
            if (b instanceof TankBehaviour) {
                TankBehaviour tb = (TankBehaviour) b;
                if (tb.mapSeedReceived) {
                    gameMap.generateWalls(tb.receivedMapSeed);
                    tb.mapSeedReceived = false;
                    return;
                }
            }
        }
    }
}
```

## 五、影响范围

- **修复覆盖的路径**: 大厅直连、CONFIG 手动输入、房间列表（保持兼容）
- **向后兼容**: 房间列表路径行为不变（之前碰巧一致，现在改为权威一致）
- **线程安全**: RPC 在 IO 线程执行，`volatile` 保证主线程可见性
- **不影响**: Sandbox 场景（纯内存，Server/Client 同进程，种子一定一致）

## 六、后续方向

- 当前 `rpcSyncMapSeed` 通过 Host 坦克广播，语义上应属于"全局同步"而非"坦克行为"。后续可引入全局 ClientRpc（不绑定特定 NetworkObject）使架构更清晰。
