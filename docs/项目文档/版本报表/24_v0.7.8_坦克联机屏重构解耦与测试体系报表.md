## 三、重构架构设计

### 3.1 重构前：上帝类（God Class）

```
NetcodeTankOnlineScreen (1116行)
 ├── 配置面板 UI 输入
 ├── 网络会话管理 (start/shutdown)
 ├── Server 游戏逻辑 (输入/移动/死亡/tick)
 ├── Client 游戏逻辑 (预测/RPC/子弹模拟)
 ├── 子弹物理 + 碰撞 (Server + Client 分散)
 ├── 地图墙体 + 碰撞 (生成/边界/推出)
 ├── 渲染 (6 种模式全在一个类)
 ├── 连接事件 + 坦克 Spawn
 └── 相机跟随
```

**核心痛点**: 无法单元测试、修一处坏三处、可读性极差、扩展困难。

### 3.2 重构后：协调器 + 4 个职责模块

```
NetcodeTankOnlineScreen (478行, 协调器)
 │
 ├── TankGameMap (139行)          ← 地图数据 + 碰撞
 │    ✅ 纯数据/数学, 可单元测试
 │
 ├── BulletSystem (184行)         ← 子弹物理 + 碰撞管理
 │    ✅ 纯逻辑, 可单元测试
 │
 ├── TankSpawnSystem (106行)      ← 出生/复活/颜色
 │    ✅ 静态工具, 可单元测试
 │
 └── TankGameRenderer (359行)     ← 全部渲染
      ⚠️ 依赖 NeonBatch/BitmapFont, 手动验证
```

Screen 精简为**网络会话管理 + 状态机 + Server/Client 帧逻辑调度**的协调器角色。


### 4.2 分层边界说明

| 层级 | 模块位置 | 依赖方向 | 可测试性 |
|------|---------|---------|---------|
| **核心框架** | `core/src/.../netcode/` | 仅依赖 Java 标准库 + libGDX math | ✅ 完全可单元测试 |
| **业务逻辑** | `examples/src/.../netcode/` | 依赖核心框架 + libGDX | ✅ 纯逻辑部分可测 |
| **渲染层** | `TankGameRenderer` | 依赖 libGDX 图形 API | ⚠️ 手动验证 |

**关键原则**: 核心框架层**零业务代码**，只提供通用的联机能力（序列化、同步、RPC、传输）；业务层利用框架构建具体玩法。两者通过 `NetworkBehaviour` 子类化和 `NetworkVariable` 声明式绑定衔接。

## 六、Bug 修复：ReceiveSequenceTracker 首包位域

### 6.1 问题

`ReliableUdpTransport.ReceiveSequenceTracker.accept()` 在接收**首个封包**时：

```java
// 修复前 (Bug)
if (highestAccepted < 0) {
    highestAccepted = seqNum;
    receivedBits[0] = true;    // ← 标记在数组头部（错误位置）
    return true;
}
```

滑动窗口的语义是 `receivedBits[WINDOW_SIZE - 1]` 对应 `highestAccepted`，而 `receivedBits[0]` 对应窗口最左端（最旧的序列号）。首包标记在 `[0]` 导致后续窗口滑动时丢失了首包的接收记录，**重复的旧包无法被正确去重**。

### 6.2 修复

```java
// 修复后
if (highestAccepted < 0) {
    highestAccepted = seqNum;
    receivedBits[WINDOW_SIZE - 1] = true;  // ← 标记在窗口末端（正确位置）
    return true;
}
```

### 6.3 影响范围

该 Bug 仅在 **Server/Client 建立连接后的前几个可靠包** 存在去重失效的风险。由于 Spawn/RPC 包均走 Reliable 通道，在极端网络环境下（如丢包导致重传）可能导致 Client 重复处理 Spawn 消息。修复后全部 11 个 ReliableUdpTransportTest 用例通过。

## 七、关键设计决策

### 7.1 `createBullet()` vs `spawnBullet()`

BulletSystem 提供两种创建接口：

| 方法 | 用途 | 是否发 RPC |
|------|------|:---------:|
| `createBullet(tank, ownerId)` | 单元测试 | ❌ |
| `spawnBullet(tank, ownerId)` | 生产环境 | ✅ `sendClientRpc` |

测试中无法绑定 `NetworkManager`，故分离纯数据创建与网络广播。

### 7.2 RPC 空值守护

所有 `sendClientRpc` / `sendServerRpc` 调用前加 `if (getManager() != null)` 判断，确保：
- 单元测试中 `TankBehaviour` 未绑定 Manager 时不会抛 `IllegalStateException`
- 不影响生产环境正常 RPC 发送

### 7.3 Renderer 持有画笔

`TankGameRenderer` 内部创建并管理 `NeonBatch` + `BitmapFont`，Screen 不再直接持有图形资源。渲染代码从 Screen 中完全剥离，Screen 只调用 `renderer.renderXxx(...)` 并传入数据。

### 7.4 TankSpawnSystem 静态工具模式

出生点/颜色/配置/复活逻辑均为 `static` 方法，坦克映射 `Map<Integer, TankBehaviour>` 仍由 Screen 持有并传入。避免了 System 需要管理网络对象生命周期的复杂度。
