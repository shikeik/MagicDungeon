## 二、BUG 1：攻击按钮有反馈但不发子弹

### 2.1 信号链路追踪

攻击按钮触发子弹的完整调用链：

```
用户触摸 VirtualButton
  │
  ▼ touchDown() — Scene2D 事件回调
  VirtualButton.touchDown()
    → InputManager.injectAction(ATTACK, true)
      → simulatedJustPressedActions.add(ATTACK)    ← ①写入
  │
  ▼ LibGDX 主循环
  ApplicationListener.render()
    → ScreenManager.render()
      → InputManager.update()                       ← ②清空！
        → simulatedJustPressedActions.clear()
      → NetcodeTankOnlineScreen.render(delta)
        → isJustPressed(ATTACK)                     ← ③读取
          → simulatedJustPressedActions.contains(ATTACK) → false!
```

### 2.2 根因：LibGDX 帧生命周期时序

**关键事实**: LibGDX 的 input 事件回调（`touchDown`/`touchUp`）在 **同一帧** 的 `render()` **之前** 被派发。

```
═══ 一帧的执行顺序 ═══
① Input Events 派发 → touchDown → injectAction(ATTACK) → pending 集合
② render() 开始
   ├─ InputManager.update() → pending.clear()  ← 信号被清空 ❌
   ├─ screen.render(delta) → isJustPressed(ATTACK) → 读 pending → 空!
   └─ render() 结束
```

时序图：

```
时间轴 ──────────────────────────────────────────────────►

   Input Event       InputManager.update()     isJustPressed(ATTACK)
   (touchDown)
       │                     │                        │
       ▼                     ▼                        ▼
  ┌──────────┐         ┌──────────┐             ┌──────────┐
  │ pending  │         │ pending  │             │ pending  │
  │ += ATTACK│  ──→    │ .clear() │    ──→      │ .contains│
  │          │         │          │             │ → false! │
  └──────────┘         └──────────┘             └──────────┘

  ◄── 同一帧内：写入后立即被清空，读取时已经丢失 ──►
```

**根因总结**: `injectAction()` 写入 `simulatedJustPressedActions`；同帧 `update()` 清空该集合；之后 `isJustPressed()` 读取同一集合 → 永远空。信号在**写入和读取之间被 update() 抹杀**。

### 2.3 为什么摇杆移动不受影响？

| 输入类型 | 注入方式 | 读取方式 | 是否受影响 |
|---------|---------|---------|-----------|
| 摇杆移动 | `injectAxis()` 写入 `injectedAxes` | `getAxis()` 读取 `injectedAxes` | ❌ 不受影响 — 轴值是持续覆盖式写入，不被 `update()` 清空 |
| 攻击按钮 | `injectAction(ATTACK, true)` → `simulatedJustPressedActions` | `isJustPressed(ATTACK)` → 检测该集合 | ✅ **受影响** — justPressed 是单帧脉冲信号，被 `update()` 清空 |

### 2.4 修复方案：双缓冲机制

引入 **ready/pending 双缓冲**，将"写入"和"读取"解耦到不同的集合：

```
═══ 修复后的帧生命周期 ═══
帧 N:
  ① Input Events → injectAction(ATTACK) → pending 集合
  ② render() 开始
     ├─ update():
     │    ready.clear()
     │    ready.addAll(pending)    ← pending 移入 ready
     │    pending.clear()          ← 清空 pending（为下帧准备）
     ├─ isJustPressed(ATTACK) → 读 ready → 包含 ATTACK ✓
     └─ render() 结束

帧 N+1:
  ① 无新 input event → pending 空
  ② render() 开始
     ├─ update(): ready = pending（空）
     ├─ isJustPressed(ATTACK) → 读 ready → 空 ✓（正确：按钮不再被按）
     └─ render() 结束
```

时序保证：

```
帧 N-1 的 Input Events ──写入──► pending
                                    │
帧 N 的 update()  ───swap───► ready ◄── 帧 N 的 isJustPressed() 读取
                                    │
帧 N 的 Input Events ──写入──► pending（新的，为帧 N+1 准备）
```

**核心性质**: `pending` 累积 input 事件，`ready` 供当前帧查询，两者交错使用，永远不会出现"写入后立即被清空再读取"的竞态。

### 2.5 代码变更 — InputManager.java (+12 / -3)

**新增字段**:

```java
private final Set<InputAction> simulatedJustPressedActions = new HashSet<>();
/** 双缓冲：update() 将 pending 移入 ready，isJustPressed 读 ready */
private final Set<InputAction> readyJustPressedActions = new HashSet<>();

private final Set<Integer> simulatedJustPressedKeys = new HashSet<>();
/** 双缓冲：同上 */
private final Set<Integer> readyJustPressedKeys = new HashSet<>();
```

**update() 方法 — 交换替代清空**:

```java
public void update() {
    // 双缓冲交换：将上一帧 input 事件累积的 justPressed 移入 ready 供本帧读取
    readyJustPressedActions.clear();
    readyJustPressedActions.addAll(simulatedJustPressedActions);
    simulatedJustPressedActions.clear();

    readyJustPressedKeys.clear();
    readyJustPressedKeys.addAll(simulatedJustPressedKeys);
    simulatedJustPressedKeys.clear();
    // ... 其余逻辑不变
}
```

**isJustPressed() — 读取 ready 缓冲**:

```java
public boolean isJustPressed(InputAction action) {
    if (inputBlocked) return false;
    if (readyJustPressedActions.contains(action)) return true;  // ← 改为读 ready
    // ... 键盘/手柄检测不变
}
```

**isKeyJustPressed() — 同理**:

```java
public boolean isKeyJustPressed(int keycode) {
    return Gdx.input.isKeyJustPressed(keycode)
        || readyJustPressedKeys.contains(keycode);  // ← 改为读 ready
}
```

## 三、BUG 2：虚拟控件视口变形

### 3.1 根因

`TankVirtualControls` 内部创建了独立的 `Stage` + `ExtendViewport`，但存在两个缺失：

1. **初始化时未同步屏幕尺寸** — `ExtendViewport` 创建后默认世界尺寸，但从未调用 `viewport.update(screenWidth, screenHeight)` 以设置实际的 glViewport 映射
2. **缩放/旋转时未转发 resize** — `NetcodeTankOnlineScreen` 没有重写 `resize()`，`TankVirtualControls.resize()` 从未被调用

**后果**:
- 触摸坐标到 Stage 世界坐标的映射偏移 → 按钮按不到
- Stage 绘制的 glViewport 与实际屏幕不匹配 → 控件拉伸变形

### 3.2 修复方案

**在 `create()` 中立即初始化视口**:

```java
virtualControls = new TankVirtualControls(new ExtendViewport(
    uiViewport.getWorldWidth(), uiViewport.getWorldHeight()));
// 立即同步视口尺寸，确保触摸坐标映射正确
virtualControls.resize(Gdx.graphics.getWidth(), Gdx.graphics.getHeight());
```

**重写 `resize()` 转发**:

```java
@Override
public void resize(int width, int height) {
    super.resize(width, height);
    if (virtualControls != null) virtualControls.resize(width, height);
}
```

`TankVirtualControls.resize()` 内部执行：
- `stage.getViewport().update(width, height, true)` — 更新 glViewport + 重置相机
- `layoutControls()` — 根据新尺寸重新定位摇杆和攻击按钮

## 六、经验总结

### 6.1 LibGDX 帧生命周期的隐含约定

LibGDX 在一帧内的执行顺序：**Input Events → render() → GL swap**。这意味着：

- `InputProcessor` 回调（`touchDown` 等）在 `render()` **之前**执行
- 如果在 `render()` 开头清空 justPressed 信号，则同帧注入的信号会被抹杀
- **原则**: 对于"瞬时脉冲"型信号（justPressed），必须采用双缓冲或延迟一帧清空

### 6.2 Scene2D Viewport 的初始化陷阱

创建 `Stage(viewport)` 后，如果不手动调用 `viewport.update(screenWidth, screenHeight)`：

- viewport 的 glViewport 参数为默认值（通常是 0×0 或上一次设置的值）
- `Stage.screenToStageCoordinates()` 计算出错误的触摸坐标
- 绘制时 `viewport.apply()` 使用错误的裁剪区域

**原则**: 新建 Stage/Viewport 后，必须在同一帧内调用 `viewport.update()` 或确保 `resize()` 会被触发。
