## 三、v0.7.6 — 可配置 Tick Rate

### 3.1 问题背景

旧版 `tick()` 每帧无条件调用，网络同步频率完全绑定帧率（~60Hz），但：
- 帧率波动时同步频率不稳定
- 无法根据带宽/场景需求调整频率

### 3.2 累加器模式

```java
// 新增 API
manager.setTickRate(60);         // 设为 60Hz（支持 1~128）
manager.tick(Gdx.graphics.getDeltaTime());  // 累加器驱动

// 内部实现
tickAccumulator += delta;
while (tickAccumulator >= tickInterval) {
    tickAccumulator -= tickInterval;
    tickInternal();  // 执行一次网络同步
}
```

- 帧率 30fps → 每帧调 2 次 tickInternal → 依然保持 60Hz 同步
- 帧率 120fps → 每 2 帧调 1 次 → 不浪费带宽
- 保留无参 `tick()` 向后兼容

## 四、v0.7.6 — 客户端预测平滑调和

### 4.1 问题背景

`NetworkVariable.deserialize()` 收到服务端值后 **硬覆盖** 本地预测值：

```
客户端预测: x = 105  →  服务端权威: x = 100  →  硬覆盖到 100  →  视觉回弹/"扯着走"
```

即使在 0 延迟回环 (127.0.0.1) 联机也能感受到明显的抖动和拉扯感。

### 4.2 解决方案 — 指数衰减插值

```java
// 启用平滑调和
networkVar.enableSmooth(
    10f,    // reconcileSpeed: 收敛速度（值越大追越快）
    50f,    // snapThreshold: 差距超过此值直接硬拉回
    0.5f    // reconcileTolerance: 误差小于此值直接吸附
);

// 每帧驱动
for (NetworkVariable<?> v : allVars) {
    v.reconcileTick(delta);
}
```

调和流程：

```
服务端值到达 → 存入 serverAuthoritativeValue（不覆盖本地值）
                    ↓
每帧 reconcileTick():
  diff = |localValue - serverValue|
  if diff > snapThreshold:  直接拉回（瞬移太远不能平滑）
  elif diff < tolerance:    吸附到服务端值（已足够接近）
  else:  localValue = lerp(localValue, serverValue, speed * delta)  ← 平滑过渡
```

### 4.3 效果

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 回环 (0ms) | 每帧明显回弹抖动 | 完全无感知 |
| 局域网 (~1ms) | 轻微扯着走 | 丝滑 |
| 公网 (~50ms) | 严重拉橡皮筋 | 平滑过渡 |

## 五、v0.7.7 — 公网崩溃根因分析

### 5.1 用户报告

> "安卓公网连 PC 房间进来无法同步任何东西了，只能生成客户端实体，并且玩了一会儿直接 allocate buffer 溢出"

### 5.2 崩溃链路还原

```
公网传输 → 偶尔收到损坏/截断 UDP 封包
              ↓
NetBuffer.readString() 读到垃圾长度 (如 0x0FFFFFFF)
              ↓
new byte[268435455] → OOM / BufferOverflowException
              ↓
onReceiveData() 无 try-catch → 未捕获异常杀死 IO 接收线程
              ↓
IO 线程死亡 → 之后60Hz状态同步包(Unreliable)全部无人接收 → "无法同步"
              ↓
ACK 停止回复 → 服务端可靠包不断重传 → PendingBuffer 膨胀
              ↓
最终 allocate buffer 溢出 💥
```

### 5.3 为什么"只能生成实体"

- Spawn 是 Reliable 包，在 IO 线程崩溃 **之前** 已经成功收到并创建了实体
- IO 线程崩溃 **之后**，所有后续的 Unreliable 状态同步包无人接收

### 5.4 排除的可能原因

| 假设 | 排除原因 |
|------|----------|
| 字节序 (Endianness) 不同 | Android 和 PC 都使用 Java `ByteBuffer`，默认 Big Endian，一致 |
| packetType 编码差异 | 两端使用相同的 `NetBuffer.writeInt()`，格式完全一致 |
| UDP Socket 绑定问题 | 握手成功 + Spawn 收到 = Socket 通信正常 |

## 六、v0.7.7 — 修复详情

### 6.1 NetBuffer 防御性安全升级

| 改动 | 说明 |
|---|---|
| 写入自动扩容 | `ensureWriteCapacity()` 按需翻倍，上限 64KB |
| `readString()` 长度上限 | `MAX_STRING_LENGTH = 4096`，超限抛 `NetBufferUnderflowException` |
| 全 read 方法前置检查 | `readInt()` / `readFloat()` / `readBoolean()` 均检查 `remaining()` |
| 自定义异常 | `NetBufferUnderflowException` 携带诊断信息（需要字节数 vs 剩余字节数） |
| `remaining()` 方法 | 新增，暴露当前缓冲区剩余可读字节数 |

### 6.2 三层 try-catch 防崩溃体系

```
层1: UdpSocketTransport.startReceiveLoop()
     └─ try { processReceivedData() } catch → 日志输出，继续接收循环

层2: ReliableUdpTransport.onRawReceive()
     └─ try { onRawReceiveInternal() } catch → 日志输出，安全忽略

层3: NetworkManager.onReceiveData()
     └─ try { onReceiveDataInternal() } catch → 区分 NetBufferUnderflowException
         和通用 Exception，日志包含 clientId + 封包长度
```

任何一层捕获到异常 → 记录诊断日志 → IO 线程继续运行 → 下一个正常包照常处理。

### 6.3 状态同步包额外校验

```java
// modifiedCount 防御（防止垃圾数据导致循环 2 亿次）
if (modifiedCount < 0 || modifiedCount > 256) {
    DLog.logErr("modifiedCount 异常: " + modifiedCount);
    return;
}

// 实体未找到时降级（Unreliable 先于 Spawn 到达是正常现象）
if (localObj == null) return;  // 不再 logErr
```

### 6.4 ReceiveSequenceTracker 滑动窗口重写

**旧实现（简单水位）**:

```
highestAccepted = max(所有已接收的 seqNum)
accept(seq): return isSeqNewer(seq, highestAccepted)

问题: seq=1 ✓ → seq=3 ✓ → seq=2 ✗ (永久丢弃！)
```

**新实现（256 位滑动窗口 + 位域去重）**:

```
receivedBits[256]: 记录最近 256 个序列号的接收状态

accept(seq):
  if seq > highest:
    滑动窗口前移，标记 seq 为已接收 → ✓
  elif highest - seq < 256:
    检查 receivedBits[对应位置]
    未接收 → 标记并接受 ✓
    已接收 → 重复包，拒绝 ✗
  else:
    太旧（超出窗口），拒绝 ✗
```

| 场景 | 旧版 | 新版 |
|------|------|------|
| 顺序到达 (1→2→3) | ✓ | ✓ |
| 乱序到达 (1→3→2) | ✗ seq=2 丢弃 | ✓ 全部接收 |
| 重复包 (1→1) | ✓ 拒绝 | ✓ 拒绝 |
| 256+ 旧包 | — | ✓ 拒绝 |
| 序列号环绕 (65535→0→1) | ✓ | ✓ |
