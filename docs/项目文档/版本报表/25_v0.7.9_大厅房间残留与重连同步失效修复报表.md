## 一、概述

本版本修复两个生产环境联机 Bug：

1. **房主 ESC 返回大厅后房间仍残留在 Presence 列表**（用户体验问题）

两个 Bug 分别涉及**大厅屏幕生命周期管理**和**可靠传输层序列号追踪器设计缺陷**。

## 二、Bug 1：房主返回大厅后房间仍残留

### 2.1 复现路径

1. 玩家 A 在大厅创建房间 → `lobbyManager.publishRoom()`
2. `replaceScreen(NetcodeTankOnlineScreen.class)` 跳转到游戏
3. 游戏中按 ESC → `NetcodeTankOnlineScreen.hide()` → 返回大厅
4. **预期**: 房间从列表中消失  
   **实际**: 房间仍显示在所有客户端的列表中，直到 WebSocket 心跳超时（~30s）才消失

### 2.2 根因分析

```
ScreenManager.replaceScreen(NetcodeTankOnlineScreen.class)
  └─ 先 dispose 旧的 NetcodeTankOnlineScreen 实例（如果存在）
  └─ 调 goScreen() → hide 当前 SupabaseLobbyScreen（压入屏幕栈）
  └─ 显示新 NetcodeTankOnlineScreen 实例
```

关键问题：`SupabaseLobbyScreen` 被 **hide（压栈）** 而非 **dispose**。其 `lobbyManager` 保持 WebSocket 连接，`publishRoom()` 发布的 Presence 状态继续存在。

当房主从游戏 ESC 返回时，`ScreenManager.popLastScreen()` 恢复大厅屏幕并调用 `show()`，但 `SupabaseLobbyScreen` **没有重写 `show()`**，因此无法在返回时执行任何清理。

### 2.3 修复方案

在 `SupabaseLobbyScreen` 中重写 `show()`：

```java
@Override
public void show() {
    super.show();
    // 从游戏屏幕返回大厅时，取消发布已创建的房间
    if (myRoomInfo != null && lobbyManager != null) {
        lobbyManager.unpublishRoom();
        myRoomInfo = null;
    }
    // 恢复建房按钮可用状态
    if (createRoomBtn != null) {
        createRoomBtn.setDisabled(false);
    }
}
```

**设计决策**: 选择在 `show()` 而非 `hide()` 中处理，因为：
- `hide()` 在跳转到游戏屏幕时也会触发，此时不应取消发布房间（游戏还在进行中）
- `show()` 仅在从游戏返回大厅时触发，此时游戏已结束，房间应当清除
- `myRoomInfo != null` 作为守卫条件，确保只有房主（发布过房间）才触发取消

### 2.4 修复后行为

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 房主创建房间 → 进入游戏 | 房间保持发布 ✅ | 房间保持发布 ✅ |
| 房主 ESC 返回大厅 | 房间残留 ~30s ❌ | 房间立即消失 ✅ |
| 客户端加入房间 → 返回大厅 | 无影响 ✅ | 无影响 ✅ (`myRoomInfo == null`) |

## 三、Bug 2：客户端断开后重连同步全失效 [严重]

### 3.1 复现路径

1. 房主创建房间，客户端 A 加入 → 正常同步 ✅
2. 客户端 A 按 ESC 断开连接（`shutdownNetwork()` → UDP 发送断开通知）
3. 客户端 A 重新加入同一房间 → 实体 Spawn 正常，但 **位置不同步、子弹不同步**
4. 实际表现：坦克可以看到，但按 WASD 不动，按 J 不开火

### 3.2 根因分析（三层问题）

#### 问题 1：全局序列号追踪器导致重连客户端的 RPC 被静默丢弃

这是 **核心致命原因**。

```
修复前架构:
┌─────────────── Server 端 ReliableUdpTransport ───────────────┐
│                                                               │
│  ReceiveSequenceTracker receiveTracker  ← 全局唯一，所有客户端共用  │
│  highestAccepted = 47                  ← 上一个客户端走到 seq=47   │
│                                                               │
│  Client A 重连后发送第一个 RPC:                                  │
│    ServerRpc("rpcMoveInput") → seq=0                           │
│                                                               │
│  receiveTracker.accept(0):                                     │
│    isSeqNewer(0, 47) → false (0 比 47 旧)                      │
│    age = seqDistance(0, 47) = 47                                │
│    47 < WINDOW_SIZE(256) → 检查 receivedBits[256-1-47=208]     │
│    receivedBits[208] = false → 接受? 否! 因为首次连接时原          │
│    客户端也用过 seq=0，位域中 [208] 可能已被标记                   │
│                                                               │
│  → ServerRpc 被当作「旧包或重复包」静默丢弃                       │
│  → 服务器永远收不到新客户端的移动/开火 RPC                        │
└───────────────────────────────────────────────────────────────┘
```

更精确地说：即使窗口位域碰巧没标记过 seq=0，新客户端连续发送 seq=0,1,2... 这些包也只有落在窗口范围内的才**可能**被接受，而任何 seq > highestAccepted 的包才会推进滑动窗口。但如果新客户端从 seq=0 开始而 highestAccepted=47，则 seq=0~47 都被视为「旧包」，高概率被去重或拒绝。只有当 seq 追上并超过 47 后才能正常工作——但这意味着**前 47 个 RPC 全部丢失**（移动/开火请求静默失败）。

**结论**: 全局共享序列号追踪器是不正确的设计。多个客户端各自维护独立的发送序列号，服务器端必须按客户端隔离追踪。

#### 问题 2：`broadcast()` 向所有历史地址（含已断开）发包

```java
// 修复前
public void broadcast(byte[] payload) {
    for (InetSocketAddress addr : clientAddresses) {  // ← 遍历所有历史地址
        sendWithLengthPrefix(payload, addr);
    }
}
```

`clientAddresses` 是 `CopyOnWriteArrayList`，设计上不移除已断开客户端的地址（保持 clientId 索引稳定）。但 `broadcast()` 遍历整个列表，导致：
- 向已断开客户端的旧地址发送数据（浪费带宽）
- 如果旧端口被 OS 回收并分配给其他进程，可能造成数据泄露

虽然这个问题本身不是同步失效的直接原因，但修复后能减少无效网络 IO。

#### 问题 3：心跳超时清理路径未同步清理追踪器

`checkHeartbeatTimeouts()` 直接调用 `userConnectionListener.onClientDisconnected()`，绕过了连接事件拦截器，导致超时断开的客户端的追踪器不会被清理。

### 3.3 修复方案

#### 3.3.1 按 clientId 隔离序列号追踪器

```java
// 新增：Server 端按客户端隔离的追踪器
private final Map<Integer, ReceiveSequenceTracker> clientReceiveTrackers 
    = new ConcurrentHashMap<>();
```

RELIABLE 通道接收处理中，根据身份选择追踪器：

```java
case CHANNEL_RELIABLE: {
    // ...解析 seqNum, ackNum...
    
    // 选择对应的序列号追踪器
    ReceiveSequenceTracker tracker;
    if (rawTransport.isServer() && clientId >= 0) {
        tracker = clientReceiveTrackers.get(clientId);
        if (tracker == null) {
            tracker = new ReceiveSequenceTracker();
            clientReceiveTrackers.put(clientId, tracker);
        }
    } else {
        tracker = receiveTracker; // Client 端仍用全局追踪器
    }
    
    if (!tracker.accept(seqNum)) {
        sendAck(seqNum, clientId);
        return;
    }
    // ...正常处理...
}
```

#### 3.3.2 拦截连接事件管理追踪器生命周期

```java
@Override
public void setConnectionListener(NetworkConnectionListener listener) {
    this.userConnectionListener = listener;
    rawTransport.setConnectionListener(new NetworkConnectionListener() {
        @Override
        public void onClientConnected(int clientId) {
            // 新客户端连入 → 创建独立追踪器（seq 从 0 开始，不受旧客户端影响）
            clientReceiveTrackers.put(clientId, new ReceiveSequenceTracker());
            if (listener != null) listener.onClientConnected(clientId);
        }
        @Override
        public void onClientDisconnected(int clientId) {
            // 客户端断开 → 清理追踪器（释放内存 + 确保下次重连从干净状态开始）
            clientReceiveTrackers.remove(clientId);
            clientLastRecvTimeMs.remove(clientId);
            if (listener != null) listener.onClientDisconnected(clientId);
        }
    });
}
```

#### 3.3.3 广播改为仅发活跃客户端

```java
@Override
public void broadcast(byte[] payload) {
    if (!isServerIdentity) return;
    for (int clientId : activeClientIds) {
        if (clientId >= 0 && clientId < clientAddresses.size()) {
            sendWithLengthPrefix(payload, clientAddresses.get(clientId));
        }
    }
}
```

`activeClientIds` 使用 `ConcurrentHashMap.newKeySet()`，线程安全，可在主线程安全迭代。

#### 3.3.4 心跳超时路径同步清理

```java
for (int cid : timedOut) {
    clientLastRecvTimeMs.remove(cid);
    clientReceiveTrackers.remove(cid);   // ← 新增
    rawTransport.deactivateClient(cid);
    // ...触发 userConnectionListener...
}
```

#### 3.3.5 disconnect() 全量清理

```java
@Override
public void disconnect() {
    rawTransport.disconnect();
    pendingBuffer.clear();
    clientReceiveTrackers.clear();  // ← 新增
}
```

### 3.4 修复后数据流

```
修复后架构:
┌─────────────── Server 端 ReliableUdpTransport ───────────────┐
│                                                               │
│  clientReceiveTrackers:                                       │
│    clientId=0 → Tracker { highestAccepted=47 }  ← Client A v1 │
│                                                               │
│  Client A 断开:                                                │
│    onClientDisconnected(0)                                     │
│    → clientReceiveTrackers.remove(0)  ← 清理旧追踪器            │
│                                                               │
│  Client A 重连 (获得 clientId=1):                               │
│    onClientConnected(1)                                        │
│    → clientReceiveTrackers.put(1, new Tracker())  ← 全新追踪器  │
│                                                               │
│  Client A 发送第一个 RPC: ServerRpc("rpcMoveInput") → seq=0    │
│    tracker = clientReceiveTrackers.get(1)                      │
│    tracker.highestAccepted = -1  ← 首包，直接接受 ✅             │
│                                                               │
│  → ServerRpc 正常推送给 NetworkManager                          │
│  → 服务器正确处理移动输入                                       │
└───────────────────────────────────────────────────────────────┘
```

### 3.5 线程安全分析

| 数据结构 | 类型 | 写入线程 | 读取线程 | 安全性 |
|---------|------|---------|---------|--------|
| `clientReceiveTrackers` | `ConcurrentHashMap` | IO 线程 (put/remove) | IO 线程 (get) | ✅ |
| `activeClientIds` | `ConcurrentHashMap.newKeySet()` | IO 线程 (add/remove) | 主线程 (broadcast 迭代) | ✅ 弱一致迭代 |
| `clientAddresses` | `CopyOnWriteArrayList` | IO 线程 (add) | 主线程 (broadcast get) | ✅ |
| `ReceiveSequenceTracker` 内部 | 非线程安全 | IO 线程 exclusive | IO 线程 exclusive | ✅ 单线程访问 |

每个 `ReceiveSequenceTracker` 实例仅由 IO 接收线程通过 `onRawReceive()` 访问，不存在跨线程竞争。

**核心收获**: `ReceiveSequenceTracker` 全局共享是一个隐蔽的设计缺陷——单客户端测试时完全正常，仅在**客户端断开重连**这一特定生命周期序列中才暴露。本次修复通过按 clientId 隔离追踪器，从根本上消除了多客户端序列号空间污染问题。
