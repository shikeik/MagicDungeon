## 二、问题分析：两套重复逻辑

### 2.1 重构前的代码对比

Sandbox 和 Online 各自内联了一套几乎相同的游戏逻辑：

| 逻辑模块 | SandboxScreen（内联） | OnlineScreen（内联） |
|----------|----------------------|---------------------|
| 坦克移动 | 直接 `setValue(x + dx * speed)` | `updateHostTankInput` + 远程 `pendingMove` 消费 |
| 开火 | 内联 `spawnBullet()` | 内联 `bulletSystem.spawnBullet()` |
| 死亡/复活 | 内联倒计时 + 硬编码复活位置/颜色 | 委托 `TankSpawnSystem.respawnTank()` |
| 子弹碰撞 | 手写 Iterator + `Math.hypot` | 委托 `bulletSystem.updateServerBullets()` |
| 墙体碰撞 | **无**（沙盒没有墙） | 内联 `clampToBoundary` + `pushOutOfWalls` |
| 归一化 | `private normalizeDir()` | `private normalizeDir()`（重复定义） |
| 客户端调和 | `private reconcileClientVariables()` | 内联 `reconcileTick` 循环 |

**核心问题**：修复 Online 的 bug 时容易忘记同步到 Sandbox，使得"本地快速测试"验出的行为与真实联机不一致。

### 2.2 不统一的代价（已发生过的真实问题）

- Sandbox 没有 `drainPendingBulletEvents()`，导致客户端看不到子弹
- Sandbox 没有调用 `reconcileTick()`，导致客户端位置不同步
- Sandbox 使用硬编码出生坐标，Online 使用 `TankSpawnSystem`，两边出生逻辑不同
- Sandbox 没有 `normalizeDir()`，斜向移动速度 √2 倍

## 三、解决方案：TankGameLogic 共享逻辑层

### 3.1 架构设计

```
┌─────────────────────┐     ┌──────────────────────┐
│ NetcodeTankSandbox   │     │ NetcodeTankOnline     │
│ Screen               │     │ Screen                │
├─────────────────────┤     ├──────────────────────┤
│ ① 读取输入           │     │ ① 读取输入            │
│   → pendingMoveX/Y  │     │   → pendingMove/Fire  │
│   → pendingFire     │     │   → ServerRpc 上报     │
│ ② gameLogic         │     │ ② gameLogic           │
│   .serverTick()     │     │   .serverTick()       │
│ ③ manager.tick()     │     │ ③ manager.tick()      │
│ ④ TankGameLogic      │     │ ④ TankGameLogic       │
│   .clientReconcile() │     │   .clientReconcile()  │
│ ⑤ gameLogic          │     │ ⑤ gameLogic           │
│   .clientBulletTick()│     │   .clientBulletTick() │
│ ⑥ 渲染               │     │ ⑥ 渲染                │
└─────────┬───────────┘     └──────────┬───────────┘
          │                            │
          └────────── 共享 ────────────┘
                       │
          ┌────────────▼────────────┐
          │      TankGameLogic      │
          │ (198行，纯逻辑，无渲染) │
          ├─────────────────────────┤
          │ serverTick(delta, tanks) │
          │  ├─ serverTickTanks()    │
          │  ├─ serverTickBullets()  │
          │  └─ serverTickCollision()│
          │ clientReconcileTick()    │
          │ clientBulletTick()       │
          │ normalizeDir()           │
          │ disableSmoothForLocal()  │
          └─────────────────────────┘
```

### 3.2 统一输入约定

所有坦克的输入（无论来源）都统一为 `pendingMoveX/Y` + `pendingFire` 两个瞬态字段：

| 场景 | 输入来源 | 写入方式 |
|------|---------|---------|
| Sandbox P1 | 本地 WASD | `readP1Input()` 直接写 |
| Sandbox P2 | 本地方向键 | `readP2Input()` 直接写 |
| Online Host | 本地 WASD | `readHostInput()` 直接写 |
| Online Remote | ServerRpc | `rpcMoveInput()` 写入 pending* |

**消费端完全统一**：`TankGameLogic.serverTickTanks()` 遍历所有坦克，消费 `pendingMove` 执行移动 → 消费 `pendingFire` 执行开火 → 处理死亡/复活。无论是 Host 还是 Remote，走同一条代码路径。

### 3.3 Sandbox 重写要点

- 删除全部内联的移动/子弹/碰撞/死亡/复活代码
- 改用 `BulletSystem` + `TankGameMap` + `Map<Integer, TankBehaviour>` 结构（与 Online 一致）
- 出生逻辑统一使用 `TankSpawnSystem.spawnTankForOwner()`
- render 方法只剩: 读输入 → serverTick → tick → clientReconcile → clientBullet → 画

### 3.4 Online 重构要点

- `updateServerLogic()` 从 50+ 行缩减为: 心跳检测 → `readHostInput()` → `gameLogic.serverTick()` → tick → tickReliable
- `updateHostTankInput()` 改为 `readHostInput()`：只写 `pendingMove/pendingFire`，不再直接 `setValue`
- `updateClientLogic()` 使用 `TankGameLogic.clientReconcileTick()` + `clientBulletTick()`
- 删除重复的 `normalizeDir()` 和 `respawnTank()` 包装方法
- `CLIENT_PREDICT_SPEED` 常量改为引用 `TankGameLogic.MOVE_SPEED`

## 四、嵌墙修复：本地玩家关闭位置插值

### 4.1 嵌墙根因分析

```
时间线:
  t=0  Client 在 (100, 100)，墙在 (120, 80~130)
  t=1  Client 预测移动到 (140, 100)
  t=2  Server 计算: 移动后 (140,100) 嵌入墙体 → pushOutOfWalls → 修正为 (119, 100)
  t=3  Server 权威值 (119,100) 到达 Client
  
  【插值模式】(旧行为):
  t=3~t=10  value 从 (140,100) 沿直线插值向 (119,100)
            中间路径经过墙体内部 → 视觉上坦克"嵌入墙中"再滑出
  
  【直接覆盖】(新行为):
  t=3  value 立刻变为 (119,100)
  t=4  Client 预测从 (119,100) 继续，不再经过墙体内部
```

### 4.2 修复策略：本地关插值，远程保留

| 对象 | 插值策略 | 原因 |
|------|---------|------|
| 本地玩家 x/y | **关闭**（`disableSmooth`） | 客户端预测 + 直接覆盖，服务端修正值已 pushOutOfWalls，不会嵌墙 |
| 远程玩家 x/y | **保留**（`enableSmooth`） | 远程玩家没有本地预测，直接覆盖会卡顿，插值让移动看起来流畅 |
| 所有玩家 rot | 无插值 | 旋转不涉及物理碰撞，直接覆盖即可 |

### 4.3 实现位置

`NetcodeTankOnlineScreen.updateClientLogic()` 中，每帧检测到本地玩家后调用：

```java
TankGameLogic.disableSmoothForLocalPlayer(myTank);
```

该方法内部：
```java
public static void disableSmoothForLocalPlayer(TankBehaviour tank) {
    tank.x.disableSmooth();
    tank.y.disableSmooth();
}
```

在 `NetworkVariable.disableSmooth()` 后，`deserialize()` 不再写入 `serverAuthoritativeValue` 缓冲区，而是直接覆盖 `value`，`reconcileTick()` 跳过（`smoothReconciliation=false`）。

## 六、技术决策记录

### Q: 为什么 Sandbox 也需要 `TankGameMap`？

重构前 Sandbox 没有地图和墙体，Server 端的 `pushOutOfWalls` + `clampToBoundary` 逻辑在 Sandbox 中缺失。`TankGameLogic.serverTickCollision()` 统一调用这些方法，要求 Sandbox 也持有 `TankGameMap` 实例。Sandbox 的地图尺寸设为半屏宽 × 屏幕高，暂不生成墙体（`generateWalls` 未调用），`pushOutOfWalls` 在空墙列表时原样返回，不影响性能。

### Q: 为什么 Host 坦克也走 `pendingMove` 路径？

重构前 Host 坦克在 `updateHostTankInput()` 中直接 `setValue()`，跳过了 `serverTickTanks()` 的统一消费流程。这意味着 Host 坦克的移动不经过 `serverTickCollision()` 之前的统一 pending 消费，与远程坦克走不同的代码路径。改为写入 `pendingMove` 后，所有坦克（Host/Remote/Sandbox P1/P2）完全走同一条执行路径，任何逻辑修改只需改一处。

### Q: `disableSmoothForLocalPlayer` 每帧调用是否有性能问题？

`disableSmooth()` 只是将 `smoothReconciliation` 布尔字段设为 `false`，无分配、无副作用。每帧调用的开销可忽略。更好的方案是在 Spawn 时一次性调用，但当前 Client 端的 Spawn 由 SpawnPacket 自动触发，没有明确的"本地玩家初始化"回调点，放在 update 中更安全。

## 七、后续优化方向

1. **SpawnCallback**：给 `NetworkManager` 增加 `onLocalPlayerSpawned(TankBehaviour)` 回调，在其中一次性调用 `disableSmoothForLocalPlayer`，避免每帧检查
2. **Sandbox 墙体测试**：Sandbox 可选生成墙体（调用 `gameMap.generateWalls(seed)`），覆盖墙体碰撞场景的快速验证
3. **客户端预测+碰撞**：本地预测移动后也执行 `pushOutOfWalls`，减少服务端修正幅度和视觉跳变
4. **远程玩家插值+碰撞感知**：插值路径不走直线，而是沿服务端验证过的路径回放（需要额外的路径数据下发，复杂度较高）
