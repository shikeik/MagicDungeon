## 一、概述

本轮迭代将 Supabase 云大厅从 **REST 表操作 + HTTP 轮询 + 心跳 PATCH** 架构，彻底迁移到 **Realtime Presence (WebSocket + Phoenix Channel)** 架构。核心收益是：离线检测从 30~60 秒降至毫秒级、数据库 I/O 归零、代码量减半。

同时打通了大厅屏幕到游戏屏幕的完整链路——建房/加入不再是空壳操作，而是携带 IP/Port 直接跳转 `NetcodeTankOnlineScreen` 启动 UDP 服务。

## 二、架构变更

### 2.1 新旧方案对比

```
旧方案 (v0.7.1 REST 表操作)          新方案 (v0.7.2 Realtime Presence)
============================          ====================================
建房    POST /rooms                   channel.track({房间信息})
心跳    Timer 15s → PATCH last_ping   WebSocket 自带 (phx_heartbeat 30s)
刷新    Timer 10s → GET /rooms        presence_state/diff 事件自动推送
幽灵    pg_cron 每分钟 DELETE          WebSocket 断连即消失 (毫秒级)
销毁    DELETE /rooms?id=eq.xxx       channel.untrack() 或直接断连
DB负载  高 (持续 HTTP 读写)             零 (纯内存状态)
```

```
┌──────────────────────────────────────────────┐
```

## 四、核心技术实现细节

### 4.1 PhoenixChannel — 协议层

Supabase Realtime 使用 **Phoenix Channel 协议** (基于 WebSocket)：

1. **连接**: `wss://{PROJECT_REF}.supabase.co/realtime/v1/websocket?apikey={KEY}&vsn=1.0.0`
2. **加入频道**: 发送 `phx_join` 消息，payload 中携带 Presence key，服务端回复 `status: "ok"`
3. **心跳**: 每 30 秒发送 `heartbeat` 到 `phoenix` topic，防止连接被服务器断开
4. **Presence Track**: 发送 `{ type: "presence", event: "track", payload: {房间数据} }` 到频道 topic
5. **实时推送**: 服务端推送 `presence_state` (全量) 和 `presence_diff` (增量 joins/leaves)
6. **自动重连**: WebSocket 断连后指数退避重连 (1s, 2s, 4s... 最大 30s，最多 10 次)

### 4.2 PresenceLobbyManager — 业务层

- 内部维护 `Map<String, PresenceRoomInfo> presenceState` 缓存，根据 `presence_state` 和 `presence_diff` 事件增量更新
- 通过 `OnRoomsSyncListener.onSync(List<PresenceRoomInfo>)` 将全量房间列表推送给 UI 层
- `publishRoom()` 幂等——重复调用会覆盖之前的状态（用于更新人数/状态）
- 所有回调已通过 `Gdx.app.postRunnable()` 切换到 LibGDX 主线程

### 4.3 大厅到游戏的完整链路

```
建房: 输入昵称 → PublicIPResolver.resolvePublicIP()
      → PresenceLobbyManager.publishRoom({name, ip, port, 1, 6})
      → NetcodeTankOnlineScreen.preConfigureAsHost(port)
      → ScreenManager.goScreen() → create() 检测预配置 → startNetwork() [Host模式]

加入: 点击房间行 → 读取 PresenceRoomInfo.hostIp + hostPort
      → NetcodeTankOnlineScreen.preConfigureAsClient(ip, port)
      → ScreenManager.goScreen() → create() 检测预配置 → startNetwork() [Client模式]
```

### 5.3 已修复的测试基础设施问题

- **GdxTestRunner.postRunnable 空操作**: Proxy Mock 的 `postRunnable(Runnable)` 原先返回默认值不执行。WebSocket 回调从后台线程通过 `postRunnable` 派发，导致测试中 Latch 永远不 countDown。修复为立即同步执行传入的 Runnable，测试环境下等价于主线程直接调用。
